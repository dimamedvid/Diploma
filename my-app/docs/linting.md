# Налаштування linting у проєкті

## Обраний інструмент

Для статичного аналізу коду в проєкті було обрано **ESLint**.

## Причини вибору

ESLint є одним із найпоширеніших інструментів статичного аналізу для JavaScript-проєктів. Він дозволяє:
- виявляти типові помилки ще до запуску програми;
- підтримувати єдиний стиль коду;
- підвищувати читабельність та підтримуваність;
- спрощувати code review;
- автоматично виправляти частину проблем.

## Чому це важливо для мого проєкту

У межах бакалаврської роботи розробляється вебзастосунок, який складається з клієнтської частини `my-app` та серверної частини `server`. Тому найбільш важливими аспектами якості коду є:
- єдність стилю коду;
- зменшення кількості типових помилок JavaScript;
- підтримуваність frontend і backend частин;
- підвищення надійності взаємодії між клієнтом і сервером.

## Налаштовані правила

У проєкті використано такі базові правила:
- обов’язкове використання крапки з комою;
- використання подвійних лапок;
- обов’язкове використання фігурних дужок;
- заборона нестрогих порівнянь `==`, дозволено тільки `===`;
- попередження про невикористані змінні;
- контроль базового стилю коду.

## Ігноровані файли та директорії

Лінтер не перевіряє:
- `node_modules/`
- `build/`
- `dist/`
- `coverage/`
- `public/`
