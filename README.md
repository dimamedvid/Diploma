# Інформаційна вебсистема для створення, перегляду та оцінювання авторських літературних творів

## Опис проєкту

Цей проєкт є бакалаврською роботою, присвяченою розробці інформаційної вебсистеми для роботи з авторськими літературними творами.

Система орієнтована на користувачів, які хочуть переглядати, знаходити, оцінювати та в подальшому публікувати власні літературні твори в межах єдиної вебплатформи.

Проєкт складається з двох основних частин:

- **my-app** — клієнтська частина застосунку;
- **server** — серверна частина та API авторизації.

Архітектура побудована за модульним підходом, що спрощує підтримку, розвиток і документування коду.

---

## Мета проєкту

Метою проєкту є створення зручної та масштабованої вебсистеми для взаємодії авторів, читачів і модераторів, яка забезпечує:

- реєстрацію та авторизацію користувачів;
- перегляд і пошук літературних творів;
- фільтрацію творів за жанром та рейтингом;
- рольову організацію доступу до функцій системи;
- основу для подальшого розширення функціоналу модерації, оцінювання, коментарів і керування публікаціями.

## Архітектурні особливості

### Frontend

Frontend побудований як React-застосунок із поділом на сторінки, компоненти, сервіси та сховище стану.

Основні модулі:

- `App.js` — описує маршрути застосунку;
- `authService.js` — відповідає за HTTP-запити авторизації;
- `authSlice.js` — зберігає стан авторизації та асинхронні дії;
- `ProtectedRoute.js` — обмежує доступ до приватних сторінок;
- `HomePage.js` — реалізує пошук і фільтрацію списку творів;
- `Header.js` і `Footer.js` — формують спільний каркас інтерфейсу;
- `WorkCard.js` та `WorksGrid.js` — відповідають за відображення творів.

### Backend

Backend реалізовано на Node.js та Express.

Основні модулі:

- `index.js` — ініціалізація сервера та підключення маршрутів;
- `auth.routes.js` — маршрути реєстрації, входу та отримання даних користувача;
- `auth.middleware.js` — middleware перевірки JWT;
- `jwt.js` — генерація та перевірка токенів;
- `fileDb.js` — робота з локальним JSON-сховищем користувачів;
- `users.json` — файл для збереження даних користувачів.

### Типові проблеми при першому запуску

#### Помилка `npm is not recognized`

Причина: Node.js або npm не встановлені, або не додані в системний `PATH`.

Рішення: перевстановити Node.js LTS і повторно перевірити команди:

```bash
node -v
npm -v
```

#### Помилка `EADDRINUSE`

Причина: порт `3000` або `4000` уже використовується іншим процесом.

Рішення: завершити процес, який займає порт, або змінити порт запуску.

#### Помилка через відсутність `users.json`

Причина: відсутній файл локального сховища користувачів.

Рішення: створити файл `server/data/users.json` і записати в нього:

```json
[]
```

#### Frontend запускається, але не працюють запити до backend

Причина: backend не запущений або працює на іншому порту.

Рішення: перевірити, що сервер запущений і доступний за адресою `http://localhost:4000`.

## Документування коду

У проєкті використовується підхід документування на основі **JSDoc**.

### Що потрібно документувати

Документуванню підлягають:

- усі публічні функції;
- React-компоненти;
- Redux thunk-и, reducers і slice-и;
- middleware та серверні маршрути;
- утиліти для роботи з токенами, файлами та API;
- складна бізнес-логіка;
- нетривіальні алгоритми фільтрації, валідації або авторизації;
- взаємодія між frontend і backend;
- архітектурні рішення, які важливі для розуміння структури системи.

### Як потрібно документувати

Для кожного нового публічного елемента коду потрібно додавати JSDoc-коментар, який містить:

- короткий опис призначення;
- параметри (`@param`);
- повернене значення (`@returns`);
- можливі помилки (`@throws`), якщо застосовно;
- додаткові пояснення щодо побічних ефектів або особливостей використання.

### Правила для команди

1. Кожна нова функція, компонент або middleware мають бути задокументовані одразу під час розробки.
2. Якщо змінюється логіка функції, документація також повинна бути оновлена.
3. Якщо модуль взаємодіє з API, потрібно описувати формат вхідних і вихідних даних.
4. Якщо код містить складну логіку, крім JSDoc слід додавати короткі пояснювальні коментарі всередині реалізації.
5. Документація має залишатися актуальною після кожного коміту.
6. Усі учасники проєкту повинні дотримуватися єдиного стилю документування.
