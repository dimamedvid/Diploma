# Архітектурні рішення та взаємодія компонентів

## Загальна архітектура

Проєкт складається з двох основних частин:

- `my-app` — клієнтська частина на React;
- `server` — серверна частина на Node.js та Express.

Frontend відповідає за відображення інтерфейсу, маршрутизацію, роботу з формами, пошук, фільтрацію та керування станом авторизації.

Backend відповідає за API авторизації, перевірку даних користувача, хешування паролів, генерацію JWT-токенів і захист приватних маршрутів.

---

## Архітектурні рішення frontend

Frontend побудований за компонентним підходом.

Основні частини:

- `App.js` — маршрутизація і загальний layout;
- `Header.js`, `Footer.js` — спільні елементи інтерфейсу;
- `HomePage.js` — сторінка перегляду та фільтрації творів;
- `LoginPage.js`, `RegisterPage.js` — сторінки авторизації;
- `CabinetPage.js` — приватна сторінка користувача;
- `ProtectedRoute.js` — контроль доступу до захищених сторінок;
- `authSlice.js` — керування станом авторизації;
- `authService.js` — взаємодія з backend API.

---

## Архітектурні рішення backend

Backend побудований за модульним підходом.

Основні частини:

- `index.js` — ініціалізація Express-сервера;
- `auth.routes.js` — маршрути реєстрації, входу та отримання поточного користувача;
- `auth.middleware.js` — перевірка JWT-токена;
- `jwt.js` — генерація і перевірка JWT;
- `fileDb.js` — файлове сховище користувачів.

---

## Бізнес-логіка

У системі реалізовано такі бізнес-правила:

- логін повинен містити від 4 до 25 символів і складатися з латинських літер або цифр;
- email має бути у валідному форматі;
- пароль повинен містити 8–20 символів, принаймні одну літеру і одну цифру;
- логін і email повинні бути унікальними;
- пароль не зберігається у відкритому вигляді, а хешується;
- доступ до маршруту `/api/auth/me` дозволяється лише з валідним JWT-токеном;
- доступ до сторінки `/cabinet` на frontend дозволяється лише авторизованому користувачу.

## Важкі для розуміння алгоритми

### Пошук і фільтрація на HomePage

На головній сторінці використано алгоритм фільтрації, який одночасно враховує:

- підрядковий пошук за назвою;
- жанр;
- мінімальний рейтинг;
- максимальний рейтинг.

Для оптимізації повторних обчислень використовується `useMemo`.

### Відображення рейтингу в WorkCard

Компонент `Stars` перетворює числовий рейтинг у візуальне представлення у вигляді повних, половинних і порожніх зірок.

## Взаємодія між компонентами

Frontend і backend взаємодіють через `authService.js`, який виконує HTTP-запити до API авторизації.

Потік авторизації виглядає так:

1. користувач вводить дані у `LoginPage` або `RegisterPage`;
2. сторінка викликає Redux thunk з `authSlice.js`;
3. thunk викликає метод `authService.js`;
4. backend обробляє запит через `auth.routes.js`;
5. у разі успіху сервер повертає токен і дані користувача;
6. `authSlice.js` зберігає токен і користувача в Redux store та localStorage;
7. `ProtectedRoute.js` використовує цей стан для контролю доступу до приватних сторінок.
