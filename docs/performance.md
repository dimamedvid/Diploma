# Профілювання та оптимізація продуктивності

## 1. Мета

Метою роботи є дослідження продуктивності вебзастосунку, виявлення вузьких місць у frontend та backend частинах системи, а також підготовка до подальшої оптимізації коду.

## 2. Коротка характеристика проєкту

Проєкт складається з двох основних частин:
- `my-app` , frontend на React;
- `server` , backend на Node.js та Express.

Frontend реалізує сторінки перегляду, пошуку і фільтрації літературних творів, а також сторінки авторизації користувачів.  
Backend реалізує API для реєстрації, входу, отримання даних поточного користувача та перевірки JWT.

У поточній версії окрема СУБД не використовується. Дані користувачів зберігаються у файловому сховищі `server/data/users.json`.

## 3. Інструменти профілювання

### Frontend
- React DevTools Profiler
- Chrome DevTools Performance
- Chrome DevTools Memory

### Backend
- Node.js Inspector
- `console.time()` / `console.timeEnd()`
- `process.memoryUsage()`

### Профілювання сховища даних
Оскільки в проєкті не використовується окрема СУБД, замість профілювання SQL-запитів аналізувалась продуктивність читання і запису файлу `users.json`.

## 4. Ключові метрики продуктивності

Для frontend аналізувались:
- час відкриття головної сторінки;
- `LCP` , Largest Contentful Paint;
- `INP` , Interaction to Next Paint;
- `CLS` , Cumulative Layout Shift;
- час виконання JavaScript, `Scripting`;
- час побудови інтерфейсу, `Rendering`;
- час малювання інтерфейсу, `Painting`;
- використання пам’яті браузером;
- час рендеру окремих React-компонентів.

Для backend аналізувались:
- загальний час обробки запиту реєстрації;
- загальний час обробки запиту логіну;
- час читання `users.json`;
- час запису `users.json`;
- час хешування пароля;
- час перевірки пароля;
- час генерації JWT.

## 5. Тестові сценарії

### Frontend
1. Відкриття головної сторінки.
2. Введення пошукового запиту.
3. Зміна жанру у фільтрі.
4. Серія взаємодій із фільтрами для перевірки використання пам’яті.

### Backend
1. Успішна реєстрація нового користувача.
2. Невдалий логін.
3. Успішний логін існуючого користувача.

## 6. Методологія профілювання

Профілювання проводилось окремо для клієнтської та серверної частин системи.

На frontend досліджувались час рендеру і повторні рендери компонентів `HomePage`, `FiltersBar`, `WorksGrid`, а також витрати на фільтрацію списку творів. Для цього використовувались React DevTools Profiler, Chrome DevTools Performance та Chrome DevTools Memory.

На backend досліджувались маршрути `/api/auth/register` і `/api/auth/login`, а також файлові операції читання і запису `users.json`. Для точного вимірювання часу виконання окремих ділянок коду в серверні маршрути було додано виклики `console.time()` та `console.timeEnd()`.

Оскільки окрема БД у проєкті відсутня, базове профілювання запитів до БД було замінено аналізом продуктивності файлового сховища `users.json`.

### 7.2. React Profiler

У React DevTools Profiler було встановлено, що:
- одним із commit-ів з найбільшим часом рендеру був commit із загальним `Render = 3.4 ms`;
- причиною оновлення був компонент `HomePage`;
- найбільший внесок у рендер серед дочірніх компонентів зробив `FiltersBar`, приблизно `1.5 ms`.

Це свідчить про те, що логіка фільтрації та елементи блоку фільтрів є важливими для продуктивності frontend-частини.

### 7.3. Пам’ять frontend

Для перевірки використання пам’яті було виконано два heap snapshot-и.

| Snapshot | Використання пам’яті |
|---|---:|
| До взаємодії | 16.8 MB |
| Після взаємодії | 18.2 MB |
| Приріст | 1.4 MB |
| Приріст у % | 8.3% |

За результатами аналізу критичного або різкого зростання пам’яті не виявлено. Отриманий приріст можна пояснити накопиченням внутрішніх структур JavaScript-рушія та об’єктів, створених під час роботи сторінки. Явних ознак витоку пам’яті на цьому етапі не зафіксовано.

### 8.2. Виявлені гарячі точки

Під час базового профілювання було визначено щонайменше три гарячі точки застосунку:

1. **`FiltersBar`**  
   У React Profiler цей компонент показав найбільший внесок у commit із максимальним часом рендеру, приблизно **1.5 ms** із загального `Render = 3.4 ms`.

2. **`register_hash_password`**  
   Операція хешування пароля під час реєстрації зайняла **61.221 ms**, що становить приблизно **93.1%** загального часу маршруту реєстрації.

3. **`login_compare_password`**  
   Перевірка пароля під час успішного логіну зайняла **55.201 ms**, що становить приблизно **96.4%** загального часу маршруту логіну.

Додатково до важливих проблемних ділянок було віднесено:
- компонент `HomePage`, який ініціює оновлення інтерфейсу;
- логіку пошуку та фільтрації, оскільки в усіх frontend-сценаріях домінує `Scripting`;
- операції `readUsers()` і `writeUsers()`, які повністю читають і записують JSON-файл при кожній операції авторизації.

### 8.3. Виявлені проблеми з продуктивністю

#### Проблеми frontend-частини
- найбільше навантаження припадає на виконання JavaScript-коду під час пошуку та фільтрації;
- оновлення інтерфейсу ініціює компонент `HomePage`;
- компонент `FiltersBar` створює найбільший внесок у час рендеру серед дочірніх компонентів;
- логіка пошуку та фільтрації є дорожчою за `Rendering` і `Loading`.

#### Проблеми backend-частини
- основним bottleneck є `bcrypt.hash()` під час реєстрації;
- основним bottleneck є `bcrypt.compare()` під час логіну;
- файлове сховище `users.json` повністю читається і записується при кожній операції, що є потенційною проблемою масштабованості.

### 8.4. Підсумок аналізу результатів

Отримані результати показали, що:
- у frontend-частині головне навантаження створює JavaScript-логіка, пов’язана з пошуком, фільтрацією та оновленням стану React-компонентів;
- у backend-частині основний час витрачається на криптографічну обробку паролів;
- використання JSON-файлу замість БД поки що не створює великого навантаження, але є архітектурним обмеженням при масштабуванні;
- критичних проблем із використанням пам’яті на frontend на цьому етапі не виявлено.

## 9. Висновок

У результаті виконання першої частини лабораторної роботи було досліджено інструменти профілювання для технологій, використаних у проєкті, визначено ключові метрики продуктивності, підготовлено тестові сценарії та виконано базове профілювання frontend і backend частин застосунку.

Отримані результати показали, що:
- у frontend-частині основні витрати припадають на виконання JavaScript-коду під час роботи з пошуком і фільтрами;
- у backend-частині головними вузькими місцями є операції `bcrypt.hash()` та `bcrypt.compare()`;
- файлове сховище `users.json` поки не є критичним за часом, але є потенційною проблемою при масштабуванні;
- критичних проблем із використанням пам’яті на frontend-частині на цьому етапі не виявлено.

На основі проведеного аналізу були визначені гарячі точки, які стануть об’єктом подальшої оптимізації на наступному етапі роботи.
