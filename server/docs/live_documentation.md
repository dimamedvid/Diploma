# Жива документація на основі тестів

## Загальна ідея

У проєкті тести використовуються не лише для перевірки правильності роботи коду, але і як приклади очікуваної поведінки системи.

Такий підхід можна розглядати як "живу" документацію, оскільки тести описують реальні сценарії використання компонентів, API та користувацьких дій.

---

## Frontend

### authService.test.js

Містить приклади використання клієнтського сервісу авторизації:

- реєстрація користувача;
- обробка помилки входу;
- передача Bearer-токена в захищений запит.

### HomePage.search.test.js

Документує поведінку головної сторінки:

- початкове відображення всіх творів;
- пошук за назвою без урахування регістру;
- фільтрацію за жанром;
- фільтрацію за діапазоном рейтингу.

### ui.steps.js, world.js, hooks.js

Описують поведінку користувача на рівні BDD-сценаріїв:

- відкриття застосунку;
- перехід на сторінку входу;
- введення логіна і пароля;
- надсилання форми;
- перевірку редиректу;
- пошук літературних творів.

---

## Backend

### auth.middleware.test.js

Документує поведінку middleware авторизації:

- помилка при відсутності токена;
- помилка при недійсному токені;
- успішне зчитування payload при валідному токені.

### auth.routes.test.js

Документує правила роботи API авторизації:

- перевірку обов’язкових полів;
- перевірку логіна, email і пароля;
- конфлікти через зайнятий логін або email;
- успішну реєстрацію;
- успішний і неуспішний вхід.

### auth.integration.test.js

Документує повний цикл роботи API:

- реєстрацію нового користувача;
- відхилення дубліката;
- помилку входу з неправильним паролем;
- доступ до захищеного маршруту без токена;
- доступ до захищеного маршруту з валідним токеном.

---

## Висновок

Тести в проєкті використовуються як приклади правильного використання компонентів і API, тому вони виконують роль "живої" документації і допомагають підтримувати актуальний опис поведінки системи.
