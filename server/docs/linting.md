# Налаштування linting для server

## Обраний інструмент

Для серверної частини проєкту `server` було обрано **ESLint**.

## Причини вибору

ESLint є ефективним інструментом статичного аналізу для JavaScript-коду, тому він підходить для перевірки серверної логіки на Node.js. Використання ESLint дозволяє:
- виявляти типові помилки в коді ще до виконання програми;
- підтримувати єдиний стиль оформлення файлів;
- покращувати читабельність та підтримуваність серверної логіки;
- зменшувати кількість помилок при роботі з маршрутами, контролерами та обробкою запитів;
- автоматично виправляти частину стилістичних проблем.

Для backend-частини це особливо важливо, оскільки сервер відповідає за обробку запитів, взаємодію з даними, авторизацію, API та стабільність роботи всього застосунку.

## Які аспекти якості коду є найважливішими

Для серверної частини проєкту найбільш важливими є:
- надійність серверної логіки;
- відсутність типових помилок JavaScript;
- єдність стилю коду;
- читабельність та підтримуваність;
- зручність подальшого розширення backend-функціоналу.

## Налаштовані правила

У `server` було налаштовано такі базові правила linting:
- обов’язкове використання крапки з комою;
- використання подвійних лапок;
- обов’язкове використання фігурних дужок у блоках;
- заборона нестрогого порівняння `==`, дозволено тільки `===`;
- попередження про невикористані змінні;
- контроль відступів;
- уніфіковане оформлення ком у багаторядкових списках та об’єктах;
- контроль пробілів усередині фігурних дужок.

Для серверної частини правило `console.log` не заборонялося, оскільки під час розробки та тестування воно може використовуватися для налагодження.

## Ігноровані файли та директорії

Лінтер не перевіряє:
- `node_modules/`
- `coverage/`
- `data/`

Ці директорії не містять основного вихідного коду серверної частини проєкту або містять службові чи допоміжні дані.
